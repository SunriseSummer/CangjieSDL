# 窗口与事件循环

## 先用一句话说明

窗口拥有系统资源，事件队列把操作系统动作交给应用；每帧应先把当前事件取空，再更新状态、绘制场景和提交画面，退出时由唯一拥有者关闭窗口。

## 为什么重要

GUI 程序不是执行一次绘制就结束。操作系统会持续发送关闭、缩放、键鼠、文本和拖放事件，应用也要持续刷新画面。若一帧只取一个事件，鼠标快速移动或窗口缩放时队列会越积越多，输入看起来“慢半拍”；若更新和绘制发生在事件循环内部，同一帧可能画很多次，空闲时却完全不画；若收到 `Quit` 就在分支中关闭窗口，外层资源管理还会再次关闭。

`pollEvent()` 返回 `?UiEvent`，`None` 表示当前队列已空，不表示程序应该退出。真正退出来自 `UiEvent.Quit` 或应用自己的业务条件。窗口缩放事件携带新尺寸，但推荐调用 `refreshSize()` 让窗口对象同步逻辑尺寸，然后下一帧从 `width`/`height` 读取。

## 工作模型

把一帧看成四个阶段：收集事件、更新业务状态、根据状态绘制、提交画面。事件阶段只做轻量翻译，耗时文件读取和复杂计算不要阻塞其中；更新阶段不调用对话框或直接绘制；渲染阶段只读已经稳定的帧状态。`delay` 是降低空转的辅助，垂直同步也可能在 `present` 等待，但二者都不能替代事件处理。

下面的反例把绘制放在每个事件之后：没有事件就不刷新，事件很多又重复绘制。

```cangjie role=contrast
while (running) {
    if (let Some(event) <- window.pollEvent()) {
        handle(event)
        update()
        draw(window.renderer)
    }
}
```

稳定结构先取空队列，再各做一次更新和绘制；退出只是改变状态，资源关闭留给外层。

```cangjie role=trace
while (running) {
    var current = window.pollEvent()
    while (let Some(event) <- current) {
        running = handle(event, running)
        current = window.pollEvent()
    }
    update()
    draw(window.renderer)
    window.delay(UInt32(8))
}
```

## 选择与取舍

“有变化才重绘”适合低频工具，但仍需持续处理事件，并在状态改变、窗口暴露或缩放时标记 dirty；“每帧重绘”更简单，适合动画和游戏，代价是持续占用渲染资源。无论选择哪一种，事件队列都应及时取空。长任务可先在事件阶段记录请求，再在合适的工作线程或分段任务中完成，最终把结果送回 UI 线程；窗口、渲染器和资源销毁保持在同一 UI 线程。

多窗口需要更谨慎。当前封装的主要模型是单主窗口；窗口创建与关闭涉及 SDL 视频子系统的引用计数。不要在彼此不知道的对象中随意调用全局退出，也不要让后台线程抢着关闭窗口。若确实使用多个窗口，先用仓库的 `window_lifecycle_test` 验证“关闭一个窗口不破坏另一个”。

## 应用这个模型

阅读计算器时先找最外层循环：内层把事件取空，事件处理只修改状态，绘制每帧执行一次。阅读雷霆战机时，顺序增加时间差和状态更新，但边界相同。

排查无响应时打印每帧处理事件数和最后事件种类；排查高 CPU 时比较画面提交的垂直同步与短延时；排查退出崩溃时列出所有关闭调用，确认只有一个拥有者负责收尾。

窗口缩放也是一次完整数据流：收到 `WindowResized`，刷新逻辑尺寸，布局下一帧读取新尺寸，渲染器按新逻辑宽高进入场景。不要只修改原生窗口而继续用旧的硬编码布局，否则画面与命中坐标会分离。

## 常见误解

`None` 不是退出信号，只是“此刻没有更多事件”。`delay(16)` 也不是可靠的 60 FPS 定时器；实际帧时长还包含事件、更新、绘制和系统调度。`KeyDown` 的 repeat 参数不是持续状态，它只表示系统重复按键事件；游戏移动仍应配对处理 `KeyUp`。最后，关闭窗口不等于所有应用资源自动安全：窗口会释放内部渲染资源，但独立 Surface、Texture 或 Cursor 仍要按所有权边界管理。
